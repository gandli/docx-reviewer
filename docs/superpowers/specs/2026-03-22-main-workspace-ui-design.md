# 主工作区界面设计

**项目：** 浏览器本地离线商务文档智能处理系统  
**日期：** 2026-03-22  
**状态：** 已确认，可进入实现规划

## 1. 设计目标

主界面需要同时支持三类高频任务：

- 打开已有文档进行审阅
- 围绕模板和资料生成新文档
- 在 AI 建议辅助下修订和优化当前文档

界面必须满足三个原则：

- 文档始终是主角，不能让聊天框抢主位
- 用户看起来像是在文档上直接工作，而不是在多个割裂面板之间来回跳
- 后续增加风险检查、证据引用、修订记录和导出确认时，不需要推翻整体布局

## 2. 推荐方案

采用固定三栏结构：

- 左栏：资料区
- 中栏：主文档区
- 右栏：助手区

这是当前最适合首版的方案，因为它同时兼顾：

- 上手简单
- 开发边界清晰
- 后续扩展稳定

不采用“聊天优先”布局，也不采用“生成工作台 / 审阅工作台”两套完全分离的壳。

## 3. 三栏职责

### 3.1 左栏：资料区

左栏负责承载这次任务相关的所有材料，而不是聊天历史。

建议包含：

- 当前工作区名称
- 当前主文档
- 模板列表
- 背景资料列表
- Excel 表格资料
- 最近引用片段
- 导入入口
- 导出入口
- 工作区设置入口

设计原则：

- 用户一眼就能知道“这次工作用到了哪些材料”
- 文件切换应该发生在左栏，而不是右栏聊天区
- 左栏可以折叠，但不能消失到影响材料管理

### 3.2 中栏：主文档区

中栏是整个系统的视觉中心。

中栏默认只承载一份主文档，不在首版中拆成“双主面板”。  
用户看到的应该是一份接近真实文档的连续页面。

交互方式：

- 默认是阅读态
- 点击段落、条款、字段、表格单元后原位进入编辑态
- 编辑完成后恢复为文档展示态
- 风险标记、高亮、批注提示、修订痕迹都覆盖在这份主文档上

关键原则：

- 视觉上像“直接在文档上改”
- 技术上不直接改原文预览 DOM
- 底层始终修改结构化稿，再回渲染到中栏

这样可以同时兼顾：

- `pdf` 的原文定位能力
- `docx` 的接近原样展示
- 后续结构化导出和修订追踪

### 3.3 右栏：助手区

右栏不是单纯聊天框，而是“聊天 + 操作 + 建议”一体化区域。

建议结构：

- 顶部：任务入口
- 中部：消息流
- 底部：当前选中内容的操作区

任务入口建议包含：

- 生成
- 审阅
- 修订
- 优化

消息流承载：

- 对话消息
- 过程说明
- 引用证据
- 风险原因

操作区承载：

- 接受建议
- 拒绝建议
- 重新生成
- 应用到当前段落
- 跳转到问题位置

## 4. 首版页面结构

```text
┌──────────────┬───────────────────────────────────┬──────────────┐
│ 左栏：资料区 │ 中栏：主文档区                    │ 右栏：助手区 │
│              │                                   │              │
│ 工作区名     │ 文档标题栏                         │ 任务切换     │
│ 主文档       │ 页面内容 / 段落 / 表格            │ 消息流       │
│ 模板         │ 风险高亮 / 修订标记 / 原位编辑    │ 建议卡片     │
│ 背景资料     │                                   │ 证据 / 操作  │
│ Excel        │                                   │              │
│ 最近引用     │                                   │              │
│ 导入/导出    │                                   │              │
└──────────────┴───────────────────────────────────┴──────────────┘
```

宽度建议：

- 左栏：20% 到 24%
- 中栏：52% 到 60%
- 右栏：24% 到 28%

核心原则：

- 中栏始终最大
- 左栏和右栏都服务于中栏
- 宽度可以拖拽调整，但不能让中栏失去主位

## 5. 关键状态设计

### 5.1 空工作区

当用户还没有导入内容时：

- 左栏显示导入引导
- 中栏显示空状态说明和建议操作
- 右栏显示可执行任务介绍

目标是让用户第一次进入时知道下一步怎么做。

### 5.2 已载入文档

当主文档已打开时：

- 左栏高亮当前主文档
- 中栏展示文档正文
- 右栏进入当前任务上下文

这是最常见的工作态。

### 5.3 选中内容

当用户点击某段或某条款时：

- 中栏高亮当前节点
- 右栏切到该节点的上下文
- 右栏展示针对该节点的建议、引用和操作

这一步是“文档驱动 AI”，不是“聊天驱动文档”。

### 5.4 应用建议

当用户接受某条建议时：

- 中栏立刻刷新对应位置
- 右栏记录该建议已应用
- 系统保存修订记录

用户不需要跳转到别的页面确认结果。

## 6. 交互流

### 6.1 审阅流

```text
左栏选中文档
-> 中栏展示文档
-> 右栏发起审阅
-> 右栏返回问题列表
-> 点击某问题
-> 中栏跳到对应位置并高亮
-> 用户接受/拒绝建议
```

### 6.2 生成流

```text
左栏选择模板和资料
-> 右栏发起生成
-> 中栏出现生成后的主文档
-> 用户点击段落继续微调
-> 右栏给出补写或优化建议
```

### 6.3 修订流

```text
用户在中栏点选条款
-> 右栏显示修订建议
-> 用户接受、拒绝或重生
-> 中栏原位更新
-> 系统记录修订状态
```

## 7. 为什么不采用其它方案

### 7.1 不采用聊天优先

原因：

- 商务文档任务本质上是文档工作，不是开放式聊天
- 一旦进入条款定位、风险确认、修订和导出，聊天框会变成噪音
- 它适合演示，不适合长期扩展成真正工作台

### 7.2 不采用两套完全分离工作台

原因：

- 首版开发量更大
- 页面状态切换更复杂
- 后面容易演变成两个产品，维护成本高

当前阶段更适合先用统一工作区承载所有任务。

## 8. 扩展预留

这套布局后续可以自然扩展出：

- 右栏增加“风险清单”标签页
- 右栏增加“修订历史”标签页
- 左栏增加“知识库片段”分组
- 中栏增加“仅看修订处”筛选
- 移动端或窄屏下将左栏、右栏折叠为抽屉

核心三栏关系不变。

## 9. 实现建议

实现时建议拆成三个前端 feature：

- `features/workspace-shell`
- `features/preview`
- `features/editor-draft`

并遵守以下边界：

- 左栏只负责上下文和材料管理
- 中栏只负责文档展示与原位编辑
- 右栏只负责 AI 消息、建议和操作
- 右栏不能成为新的“主页面”

## 10. 验收标准

这套主界面在首版中算完成，需要满足：

- 用户能在左栏快速切换当前主文档和资料
- 用户能在中栏直接阅读并原位编辑文档内容
- 用户能在右栏发起生成、审阅和修订任务
- 右栏点击问题后，中栏能准确跳到对应位置
- 用户接受建议后，中栏结果立即更新
- 整体流程不需要频繁切换页面或进入第二套工作台

## 11. 最终结论

首版主界面正式采用：

- 左边管材料
- 中间管文档
- 右边管 AI

其中最关键的产品判断是：

**看起来像在原文上改，但技术上不直接改原文预览层。**

这能同时保证：

- 用户体验自然
- 实现边界清晰
- 后续修订、审阅、导出都可控
