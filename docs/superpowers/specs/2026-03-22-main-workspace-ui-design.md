# 主工作区界面设计

**项目：** 浏览器本地离线商务文档智能处理系统  
**日期：** 2026-03-22  
**状态：** 已确认，可进入实现规划

## 1. 设计目标

主界面需要同时支持三类高频任务：

- 打开企业内部已有文档进行审阅
- 围绕模板和参考资料生成新文档
- 在 AI 建议辅助下修订和优化当前文档

界面必须满足三个原则：

- 文档始终是主角，不能让聊天框抢主位
- 用户看起来像是在文档上直接工作，而不是在多个割裂面板之间来回跳
- 后续增加风险检查、证据引用、修订记录和导出确认时，不需要推翻整体布局

## 2. 推荐方案

采用固定三栏结构：

- 左栏：资料区
- 中栏：主文档区
- 右栏：助手区

这是当前最适合首版的方案，因为它同时兼顾：

- 上手简单
- 开发边界清晰
- 后续扩展稳定

不采用“聊天优先”布局，也不采用“生成工作台 / 审阅工作台”两套完全分离的壳。

## 3. 风格定稿

本项目主界面的 UI/UX 风格正式定为：

- 高端产品感
- 简洁、优雅、克制
- 暖中性色
- 文档为中心
- 企业内部文档工作台语境
- 外层满屏贴边

明确不采用：

- 强后台感
- 夸张科技感
- 过度品牌化的营销页面视觉
- 写死“标书”等过窄业务命名

### 3.1 视觉气质

整体要像一套成熟、安静、可信赖的企业文档产品。

关键词：

- 米白
- 浅砂
- 深墨
- 纸张质感
- 安静留白
- 精致边界

视觉目标不是“炫”，而是“高级、稳、可长期使用”。

补充要求：

- 页面外层不做卡片式大边框
- 页面整体不保留外边距包裹感
- 主工作区应像 ChatGPT 一样贴边铺满窗口
- 中间文档本身仍保留纸张视觉

### 3.2 语言风格

界面文案采用通用企业文档语言，例如：

- 主文档
- 模板
- 参考资料
- 表格资料
- 生成
- 审阅
- 修订
- 优化

避免写成某一个垂直行业专用工具。

## 4. 三栏职责

### 4.1 左栏：资料区

左栏负责承载这次任务相关的所有材料，而不是聊天历史。

建议包含：

- 当前工作区名称
- 当前主文档
- 模板列表
- 背景资料列表
- Excel 表格资料
- 最近引用片段
- 导入入口
- 导出入口
- 工作区设置入口

设计原则：

- 用户一眼就能知道“这次工作用到了哪些材料”
- 文件切换应该发生在左栏，而不是右栏聊天区
- 左栏可以折叠，但不能消失到影响材料管理
- 左栏整体要更安静，像高级资料柜，而不是后台菜单
- 左栏应采用轻分隔，不做明显卡片堆叠

### 4.2 中栏：主文档区

中栏是整个系统的视觉中心。

中栏默认只承载一份主文档，不在首版中拆成“双主面板”。  
用户看到的应该是一份接近真实文档的连续页面。

交互方式：

- 默认是阅读态
- 点击段落、条款、字段、表格单元后原位进入编辑态
- 编辑完成后恢复为文档展示态
- 风险标记、高亮、批注提示、修订痕迹都覆盖在这份主文档上

关键原则：

- 视觉上像“直接在文档上改”
- 技术上不直接改原文预览 DOM
- 底层始终修改结构化稿，再回渲染到中栏
- 页面纸张感要明显，页边距、版心和留白都要比普通网页更讲究

这样可以同时兼顾：

- `pdf` 的原文定位能力
- `docx` 的接近原样展示
- 后续结构化导出和修订追踪

### 4.3 右栏：助手区

右栏采用连续对话主导，不做死板的任务标签页，也不做说明卡片堆叠。

建议结构：

- 顶部：上下文标题和轻量状态
- 中部：消息对话线程
- 底部：固定输入区和当前操作区

在输入区上方提供建议动作：

- 生成
- 审阅
- 修订
- 优化

消息线程承载：

- 用户消息
- 助手消息
- 过程说明
- 引用证据
- 风险原因

操作区承载：

- 接受建议
- 拒绝建议
- 重新生成
- 跳转到问题位置

语气要求：

- 专业克制
- 简短明确
- 不做情绪化表达
- 不像通用聊天机器人

视觉要求：

- 右栏整体应接近 ChatGPT 的对话区
- 消息需要区分用户与助手
- 输入框固定在底部
- 建议动作贴近输入框，而不是充当顶部主导航

## 5. 首版页面结构

```text
┌──────────────┬───────────────────────────────────┬──────────────┐
│ 左栏：资料区 │ 中栏：主文档区                    │ 右栏：助手区 │
│              │                                   │              │
│ 工作区名     │ 文档标题栏                         │ 对话线程     │
│ 主文档       │ 页面内容 / 段落 / 表格            │ 用户/助手消息│
│ 模板         │ 风险高亮 / 修订标记 / 原位编辑    │ 建议动作     │
│ 背景资料     │                                   │ 输入区 / 操作│
│ Excel        │                                   │              │
│ 最近引用     │                                   │              │
│ 导入/导出    │                                   │              │
└──────────────┴───────────────────────────────────┴──────────────┘
```

宽度建议：

- 左栏：20% 到 24%
- 中栏：52% 到 60%
- 右栏：24% 到 28%

核心原则：

- 中栏始终最大
- 左栏和右栏都服务于中栏
- 宽度可以拖拽调整，但不能让中栏失去主位
- 页面整体贴边铺满，不额外包一层大卡片

## 6. 首版优先状态

首版最优先打磨的页面状态定为：

- `工作状态`

原因：

- 这是用户停留时间最长的页面
- 生成、审阅、修订三类任务最终都会回到这里
- 它最能体现产品的真实质量，而不是只体现空状态包装

页面状态优先级：

1. 工作状态
2. 审阅结果状态
3. 空工作区

## 7. 关键状态设计

### 7.1 空工作区

当用户还没有导入内容时：

- 左栏显示导入引导
- 中栏显示空状态说明和建议操作
- 右栏显示可执行任务介绍

目标是让用户第一次进入时知道下一步怎么做。

### 7.2 已载入文档

当主文档已打开时：

- 左栏高亮当前主文档
- 中栏展示文档正文
- 右栏进入当前任务上下文

这是最常见的工作态。

### 7.3 选中内容

当用户点击某段或某条款时：

- 中栏高亮当前节点
- 右栏切到该节点的上下文
- 右栏展示针对该节点的建议、引用和操作

这一步是“文档驱动 AI”，不是“聊天驱动文档”。

### 7.4 应用建议

当用户接受某条建议时：

- 中栏立刻刷新对应位置
- 右栏记录该建议已应用
- 系统保存修订记录

用户不需要跳转到别的页面确认结果。

### 7.5 审阅结果状态

当一次审阅完成后：

- 中栏保留文档正文，不切换成另一种页面壳
- 风险点以高亮、边注或条款标记形式覆盖在文档上
- 右栏展示风险摘要和逐条处理入口

这保证“审阅结果”仍然属于同一个工作区，而不是另起一页。

## 8. 交互流

### 8.1 审阅流

```text
左栏选中文档
-> 中栏展示文档
-> 右栏发起审阅
-> 右栏返回问题列表
-> 点击某问题
-> 中栏跳到对应位置并高亮
-> 用户接受/拒绝建议
```

### 8.2 生成流

```text
左栏选择模板和资料
-> 右栏发起生成
-> 中栏出现生成后的主文档
-> 用户点击段落继续微调
-> 右栏给出补写或优化建议
```

### 8.3 修订流

```text
用户在中栏点选条款
-> 右栏显示修订建议
-> 用户接受、拒绝或重生
-> 中栏原位更新
-> 系统记录修订状态
```

### 8.4 工作区恢复流

```text
用户暂停或意外中断
-> 系统保存工作区摘要
-> 记录当前主文档、当前节点、待处理建议、最近证据
-> 下次重新进入工作区
-> 恢复上次工作摘要
-> 用户继续处理
```

恢复原则：

- 接续对象是工作区，不是某一条聊天线程
- 接续依赖结构化工作摘要，不依赖完整历史消息重放
- 恢复后用户应直接回到“可继续工作”的状态

## 9. 核心视觉规范

### 9.1 色彩

推荐用暖中性色作为主基调：

- 背景：米白、浅砂
- 分隔与边界：柔和暖灰
- 正文：深墨色
- 强调：低饱和金棕或深木色
- 风险：克制的砖红与琥珀色

原则：

- 不用大面积冷蓝
- 不用高饱和彩色按钮抢视线
- 让颜色服务于层级，而不是制造噪音

### 9.2 材质与边界

- 面板边界要柔和
- 阴影要浅，不做悬浮卡片堆叠感
- 中栏纸张卡片可以有更明显的精致阴影和圆角
- 左右栏尽量平静，避免过多独立卡片造成碎片感
- 左右栏以轻分隔为主，不做卡片墙

### 9.3 排版

- 标题需要克制但有分量
- 正文强调可读性和节奏
- 留白必须足够
- 文档区的版心宽度不能太散

## 10. 为什么不采用其它方案

### 10.1 不采用聊天优先

原因：

- 商务文档任务本质上是文档工作，不是开放式聊天
- 一旦进入条款定位、风险确认、修订和导出，聊天框会变成噪音
- 它适合演示，不适合长期扩展成真正工作台

### 10.2 不采用两套完全分离工作台

原因：

- 首版开发量更大
- 页面状态切换更复杂
- 后面容易演变成两个产品，维护成本高

当前阶段更适合先用统一工作区承载所有任务。

## 11. 扩展预留

这套布局后续可以自然扩展出：

- 右栏增加“风险清单”标签页
- 右栏增加“修订历史”标签页
- 左栏增加“知识库片段”分组
- 中栏增加“仅看修订处”筛选
- 左栏增加“继续上次工作”卡片
- 右栏增加“当前工作摘要”折叠区
- 移动端或窄屏下将左栏、右栏折叠为抽屉

核心三栏关系不变。

## 12. 实现建议

实现时建议拆成三个前端 feature：

- `features/workspace-shell`
- `features/preview`
- `features/editor-draft`

并遵守以下边界：

- 左栏只负责上下文和材料管理
- 中栏只负责文档展示与原位编辑
- 右栏只负责 AI 消息、建议和操作
- 右栏不能成为新的“主页面”

## 13. 验收标准

这套主界面在首版中算完成，需要满足：

- 用户能在左栏快速切换当前主文档和资料
- 用户能在中栏直接阅读并原位编辑文档内容
- 用户能在右栏发起生成、审阅和修订任务
- 右栏点击问题后，中栏能准确跳到对应位置
- 用户接受建议后，中栏结果立即更新
- 整体流程不需要频繁切换页面或进入第二套工作台
- 整体视觉符合“高端产品感、暖中性色、纸张感更强”的既定方向
- 页面外层贴边满屏，无大边框和外边距
- 右栏呈现为类 ChatGPT 的消息对话区
- 对话中断后，重新打开工作区时可以恢复结构化工作摘要并继续处理

## 14. 最终结论

首版主界面正式采用：

- 左边管材料
- 中间管文档
- 右边管 AI

其中最关键的产品判断是：

**看起来像在原文上改，但技术上不直接改原文预览层。**

这能同时保证：

- 用户体验自然
- 实现边界清晰
- 后续修订、审阅、导出都可控

同时，视觉与交互风格也正式定为：

- 高端产品感
- 暖中性色
- 左栏安静
- 中栏纸张感强
- 右栏连续对话
- AI 专业克制
